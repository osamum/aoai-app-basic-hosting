# 演習 6-1 : Application Gateway を介したアプリケーションの公開(HTTPS)

[演習 5-3](Ex05-3.md) で仮想ネットワーク内に閉域化されてた App Service に対して、Application Gateway を介してインターネットからアクセスできるように構成します。

![Application Gateway(HTTP) の追加構成](images/AG-HTTP-added-configure.png)

Azure Application Gateway は、Web アプリケーションに対するトラフィックを管理できる Web トラフィック (OSI レイヤー 7) ロード バランサーです。 Application Gateway では、URI パスやホスト ヘッダーなど、HTTP 要求の追加属性に基づいてルーティングを決定できます。

Azure Application Gateway が提供する機能の詳細については以下のドキュメントを参照してください。

* [**Azure Application Gateway の機能**](https://learn.microsoft.com/ja-jp/azure/application-gateway/features)

このタスクでは Application Gateway 用に新規に仮想ネットワークを追加し、この仮想ネットワークとサブネットに対して App Service ネットワーク設定でアクセスを許可するルール設定を行います。

## この演習で構成する通信の仕様について

この演習では Application Gateway と Web ブラウザーなどのクライアント間は HTTPS 通信を行います。

 Application Gateway で HTTS で使用する SSL の終端を行い、バックエンド プールとして登録した App Service とは HTTP 通信を行う構成とすると App Service の負荷を軽減でき、構成も簡単になります。
 
 ![Application Gateway をバックエンドプールと HTTP で接続](images/AG-backendpool-01.png)
 
 
 しかし、近年の Web アプリケーションの SDK やミドルウェアなどで HTTPS 通信を前提としているものが多いため、この演習では Application Gateway と App Service 間も HTTPS 通信を行う構成とします。

 Application Gateway とバックエンド プールとして登録した App Service 間で HTTPS 通信を行う場合、[Application Gateway 側にのみカスタム ドメインを設定し、App Service 側には App Service 既定のドメイン名(FQDN)でアクセスする](https://learn.microsoft.com/ja-jp/azure/application-gateway/configure-web-app?tabs=defaultdomain%2Cazure-portal)場合などは、App Service 側でカスタム ドメインを設定する必要はありません。

  ![Application Gateway をバックエンドプールと App Service 既定の FQDN と HTTPS で接続](images/AG-backendpool-02.png)
 

 しかし、以下のサポート ブログに記載されているように、Application Gateway のフロント エンドにカスタム ドメインを設定しており、バックエンドに指定した App Service 側でリダイレクトの応答が返されるシナリオでは前述の構成だと意図せず App Service の 既定の FQDN が利用されてしまい期待した動作とならない場合があります。 

 * [**Application Gateway - App Service 間のリダイレクトの問題**](https://jpaztech.github.io/blog/network/appgw-appservice-redirectissue/)
  
この問題の対処方法として上記のブログ記事では 2 つの方法が紹介されています。

1 つは [Application Gateway の書き換え規則を使い、リダイレクト URL を書き換える方法](https://jpaztech.github.io/blog/network/appgw-appservice-redirectissue/#application-gateway-rewrite)ですが、この方法では referer ヘッダーや Cookie などにもドメインが含まれていて書き換えでの対応が複雑となる場合には対応できません。

この演習では、もう 1 つの方法である [カスタム ドメインをつかう](https://jpaztech.github.io/blog/network/appgw-appservice-redirectissue/#%E3%82%AB%E3%82%B9%E3%82%BF%E3%83%A0-%E3%83%89%E3%83%A1%E3%82%A4%E3%83%B3%E3%82%92%E3%81%A4%E3%81%8B%E3%81%86)を参考に、App Service 側にもカスタム ドメインを設定し、Application Gateway と App Service 間も HTTPS 通信を行う構成とします。

 ![Application Gateway をバックエンドプールとカスタム ドメインを使用して HTTPS で接続](images/AG-backendpool-03.png)

この演習で実施する作業は以下の通りです。

**準備**

1. バックエンド検明書（.cer）のダウンロード
2. 演習用 App Service へのバックエンド証明書の追加
3. App Service プライベート エンドポイントのカスタム ドメイン用プライベート DNS ゾーンの追加

**Application Gateway の追加**

1. [Application Gateway 用サブネットの追加](#1-application-gateway-%E7%94%A8%E3%82%B5%E3%83%96%E3%83%8D%E3%83%83%E3%83%88%E3%81%AE%E8%BF%BD%E5%8A%A0)
2. [Application Gateway の作成](#2-application-gateway-%E3%81%AE%E4%BD%9C%E6%88%90)

## 準備

Application Gateway がバックエンド プールの App Service と HTTPS で通信する際必要になるバックエンド検証用証明書（.cer）をダウンロードし、演習用 App Service に追加します。

また、演習用 App Service を Application Gateway のバックエンド プールとして指定する際、カスタムドメイン名が名前解決できるように App Service のプライベートエンドポイント用のプライベート DNS ゾーンを追加します。

### 1. バックエンド検明書（.cer）のダウンロード

 App Service に設定した SSL 証明書の公開鍵部分を公開キー証明書( .cer )ファイルとしてダウンロードします。この作業は Key Vault にアップロードした証明書からバックエンド検証用証明書（.cer）をダウンロードします。

 この手順を実行するには、[演習 5-4 : App Service へのカスタムドメインの設定](Ex05-4.md) で App Service に設定した SSL 証明書が Key Vault にアップロードされている必要があります。

 なお、コマンド等を使用してすでに バックエンド検証用証明書（.cer）を抽出済みの場合は、この手順をスキップして構いません。

具体的な手順は以下のとおりです。

\[**手順**▶️\]

1. [Azure ポータル](https://portal.azure.com/)にログイン、[演習 4-1](Ex04-1.md#1--azure-key-vault-%E3%81%AE%E4%BD%9C%E6%88%90%E3%81%A8api-%E3%82%AD%E3%83%BC%E3%81%AE%E7%99%BB%E9%8C%B2) で作成した Key Vault の画面を開きます

2. 画面左側のメニューから \[オブジェクト\] - \[**証明書**\]をクリックします
   
   遷移した画面に [演習 5-4-1 Key Vault への証明のアップロード](Ex05-4.md#1-key-vault-%E3%81%B8%E3%81%AE%E8%A8%BC%E6%98%8E%E3%81%AE%E3%82%A2%E3%83%83%E3%83%97%E3%83%AD%E3%83%BC%E3%83%89) でアップロードした証明書がリストされるのでクリックします

   ![Key Vault の証明書の選択](images/KeyVault-select-pfx.png)

3. 証明書のバージョンがリストされるので、最新のバージョンをクリックします

    ![Key Vault の証明書のバージョン選択](images/KeyVault-select-pfx-ver.png)

4. 遷移した画面の上部にある \[**↓ CER 形式でダウンロード**\] ボタンをクリックします

    ![Key Vault の証明書のダウンロード](images/KeyVault-download-cer.png)

    Web ブラウザーによっては以下のような警告が表示される場合がありますが、証明書をダウンロードして任意のフォルダーに保存してください。
    
    `<証明書名>.cer はデバイスに問題を起こす可能性があります。このまま保持しますか?`

ここまでの作業でバックエンド検証用証明書（.cer）を取得することができました。

この検証用証明書は App Service と Application Gateway の両方で使用します。

<br>　　

### 2. 演習用 App Service へのバックエンド証明書の追加

前の作業でダウンロードしたバックエンド検証用証明書（.cer）を演習用 App Service に追加します。

具体的な手順は以下のとおりです。

\[**手順**▶️\]

1. [Azure ポータル](https://portal.azure.com/)で、演習用 App Service の画面を開き、画面左側のメニューから \[設定\] - \[**証明書**\]をクリックします

    \[証明書\] の画面に遷移するので \[**公開キー(.cer)**\] タブをアクティブにし、同タブ内の \[**+ 証明書の追加**\] ボタンをクリックします。

    ![App Service の証明書の追加](images/AppService-add-cer-btn.png)

2. 画面右に \[公開キー証明書の追加\] ブレードが表示されるので、各項目を以下のように設定します

    | 項目 | 設定内容 |
    |---|---|
    | CER 証明書ファイル \*| *📁 マークのボタンをクリックして、前の手順でダウンロードしたバックエンド検証用証明書（.cer）を選択* |
    | 証明書のフレンドリ名 \*| *任意の名前* |

    各項目を設定したら、画面下部の \[**追加**\] ボタンをクリックします

    ![App Service の証明書の追加](images/AppService-add-cer.png)

    \[公開キー(.cer)\] タブに戻るので、追加した証明書が表示されていることを確認します

    ![App Service の証明書の追加完了](images/AppService-cer-added.png)

ここまでの手順で演習用 App Service にバックエンド検証用証明書（.cer）を追加することができました。

<br>

### 3. App Service プライベート エンドポイントのカスタム ドメイン用プライベート DNS ゾーンの追加

演習用 App Service を Application Gateway のバックエンド プールとして指定する際、カスタムドメイン名が名前解決できるように App Service のプライベートエンドポイント用のプライベート DNS ゾーンを追加します。

具体的な手順は以下のとおりです。

\[**手順**▶️\]

1. Azure ポータル [ホーム画面](https://portal.azure.com/#home) ログインし、画面上部の \[**+**\] リソースの作成 アイコンか、表示されていない場合は画面左上のハンバーガーメニューをクリックし、\[**リソースの作成**\] をクリックします

    ![Azure リソースの作成](images/23aug_create_AzureResource.png)

2. \[**リソースの作成**\] 画面に遷移するので、検索ボックスに `dns ゾーン` と入力し検索します
   
    検索結果にまったく同じデザインのタイルが 2 つ表示されますが、どちらかがプライベート DNS ゾーンです。多くの場合、左側のタイルがプライベート DNS ゾーンなのでそちらをクリックします。
    
    ![プライベート DNS ゾーンの検索](images/find-result-privateDNS.png)

3. \[**DNS ゾーン**\] の画面に遷移するので、\[プラン\] が `Private DNS Zone` であることを確認し、\[**作成**\] ボタンをクリックします

    ![プライベート DNS ゾーンの作成](images/privateDNS-start-create.png)

4. \[**DNS ゾーンの作成**\] 画面の \[**基本**\] タブがアクティブになっているので、各項目を以下のように設定します

    | 項目 | 設定内容 |
    |---|---|
    | サブスクリプション \* | *演習用リソースを作成したサブスクリプション* |
    | リソース グループ \* | **AOAI-AppEnv-handson** |
    | 名前 \* | *`<カスタムドメイン名>` の形式で、演習用 App Service に設定したカスタムドメイン名を入力*<br>例: `www.contoso.jp` |
    | リージョン \* | *任意のリージョン* |

    各項目を設定したら、画面上部の \[**Virtual Network Links**\]  タブをクリックします。(※プライベート DNS レコードの追加はリソース作成後に行います)

    ![プライベート DNS ゾーンの基本設定](images/privateDNS-add-basic.png)

5. \[**Virtual Network Links**\] タブがアクティブになるので、同タブ内の \[**+ 仮想ネットワークを追加します**\] ボタンをクリックします

    画面右に \[仮想ネットワーク リンクを追加します\] ブレードが表示されるので、各項目を以下のように設定します。

    | 項目 | 設定内容 |
    |---|---|
    | リンク名 \* | `link-jpeast-vnet` |
    | サブスクリプション \* | *演習用リソースを作成したサブスクリプション* |
    | 仮想ネットワーク \* | \[**handson-jpeast-vnet**\] |
    | 自動登録を有効にする | *既定のまま* |

    各項目を設定したら、画面下部の \[**作成**\] ボタンをクリックします。

6. \[Virtual Network Links\] タブに戻るので、追加した仮想ネットワーク リンクが表示されていることを確認し、画面上部の \[**確認および作成**\] ボタンをクリックします

    ![仮想ネットワークリンクの追加](images/privateDNS-add-vnetLink.png)

    `検証に合格しました` と表示されるのを確認し、画面下部の \[**作成**\] ボタンをクリックします。
